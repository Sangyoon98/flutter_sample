# Study Session Protocol

이 문서는 에이전트가 매번 학습 세션을 진행할 때 지켜야 하는 규칙이다.

## 세션 시작 체크리스트

1. `docs/progress_log.md`를 읽고 현재 단계와 다음 작업을 확인한다.
2. 필요하면 `docs/study_plan.md`에서 현재 단계의 목표와 완료 기준을 확인한다.
3. 기존 코드 상태를 확인한다.
4. 이번 세션의 작은 목표를 하나만 정한다.
5. 사용자가 직접 따라 칠 수 있도록 설명과 코드를 함께 제공한다.

## 설명 방식

설명은 아래 순서로 한다.

1. 오늘 만들 것
2. 새로 배울 개념
3. 먼저 읽을 기존 코드
4. 따라 칠 코드
5. 코드가 동작하는 이유
6. 직접 바꿔볼 작은 과제
7. 오늘 배운 내용 요약

## 코드 작성 원칙

- 처음에는 복잡한 아키텍처를 강요하지 않는다.
- 학습 단계에 맞는 가장 단순한 구조로 시작한다.
- 위젯이 커지면 기능 단위로 파일을 분리한다.
- Dart/Flutter 기본기를 익히기 전에는 상태 관리 라이브러리를 서두르지 않는다.
- 새 패키지는 필요한 이유를 설명한 뒤 추가한다.
- 앱 코드 변경과 문서 변경을 같은 학습 세션의 일부로 다룬다.

## 세션 종료 체크리스트

1. 구현한 코드가 학습 목표와 맞는지 확인한다.
2. 가능하면 `flutter analyze`를 실행한다.
3. 가능하면 `flutter test`를 실행한다.
4. `docs/progress_log.md`에 날짜, 작업, 배운 내용, 다음 작업을 기록한다.
5. 새로 배운 핵심 개념은 `docs/learning_notes.md`에 누적한다.
6. 코드와 문서를 함께 커밋한다.

## 커밋 원칙

- 커밋은 학습 단위로 작게 만든다.
- 앱 코드만 바꾸고 문서를 빼먹지 않는다.
- 문서만 바꾼 경우에도 왜 바꿨는지 커밋 메시지에 드러낸다.

커밋 메시지 예시:

- `docs: 플러터 학습 계획 추가`
- `feat: 홈 피드 화면 뼈대 추가`
- `refactor: 게시글 카드 위젯 분리`
- `test: 게시글 좋아요 토글 테스트 추가`

## 다음 요청 처리 규칙

사용자가 아래처럼 말하면 이 문서를 기준으로 이어간다.

- "오늘 공부하자"
- "다음 작업 해줘"
- "어제 하던 거 이어서"
- "이번 단계 설명해줘"
- "커밋하자"

이때 에이전트는 먼저 진행 로그를 확인하고, 문서에 기록된 다음 작업에서 출발한다.
