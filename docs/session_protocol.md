# Study Session Protocol

이 문서는 에이전트가 매번 학습 세션을 진행할 때 지켜야 하는 규칙이다.

## 세션 시작 체크리스트

1. `docs/progress_log.md`를 읽고 현재 단계와 다음 작업을 확인한다.
2. 필요하면 `docs/study_plan.md`에서 현재 단계의 목표와 완료 기준을 확인한다.
3. 기존 코드 상태를 확인한다.
4. 이번 세션의 작은 목표를 하나만 정한다.
5. 사용자가 직접 따라 칠 수 있도록 설명과 코드를 함께 제공한다.
6. 사용자가 명시적으로 "직접 수정해줘", "적용해줘", "구현해줘"라고 요청하지 않았다면 앱 코드를 수정하지 않는다.

## 설명 방식

설명은 아래 순서로 한다.

1. 오늘 만들 것
2. 새로 배울 개념
3. 먼저 읽을 기존 코드
4. 따라 칠 코드
5. 코드가 동작하는 이유
6. 직접 바꿔볼 작은 과제
7. 오늘 배운 내용 요약

## 코드 작성 원칙

- 학습 세션의 기본 모드는 "에이전트가 설명하고, 사용자가 직접 코드를 따라 친다"이다.
- 에이전트는 먼저 수정할 위치, 지울 코드, 새로 입력할 코드를 안내한다.
- 사용자의 명시적인 요청 없이 `lib/`, `test/`, 플랫폼 코드 같은 앱 소스 파일을 직접 수정하지 않는다.
- 문서 갱신, 진행 기록 정리처럼 사용자가 요청한 관리 작업만 직접 수정할 수 있다.
- 사용자가 코드를 따라 친 뒤 오류가 나면, 에이전트는 오류 메시지를 보고 원인과 수정 코드를 안내한다.
- 처음에는 복잡한 아키텍처를 강요하지 않는다.
- 학습 단계에 맞는 가장 단순한 구조로 시작한다.
- 위젯이 커지면 기능 단위로 파일을 분리한다.
- Dart/Flutter 기본기를 익히기 전에는 상태 관리 라이브러리를 서두르지 않는다.
- 새 패키지는 필요한 이유를 설명한 뒤 추가한다.
- 앱 코드 변경과 문서 변경을 같은 학습 세션의 일부로 다루되, 앱 코드 변경은 사용자가 직접 작성하는 것을 기본으로 한다.

## 금지된 진행 방식

- 사용자가 학습을 시작하자고만 했는데 에이전트가 앱 코드를 바로 수정하지 않는다.
- 사용자의 연습 기회를 빼앗는 방식으로 완성본을 먼저 적용하지 않는다.
- 따라 칠 코드와 설명 없이 파일을 패치하지 않는다.
- 테스트나 분석을 실행하기 위해 필요한 경우에도 먼저 사용자가 코드를 입력했는지 확인한다.

## 세션 종료 체크리스트

1. 구현한 코드가 학습 목표와 맞는지 확인한다.
2. 가능하면 `flutter analyze`를 실행한다.
3. 가능하면 `flutter test`를 실행한다.
4. `docs/progress_log.md`에 날짜, 작업, 배운 내용, 다음 작업을 기록한다.
5. 새로 배운 핵심 개념은 `docs/learning_notes.md`에 누적한다.
6. 코드와 문서를 함께 커밋한다.

## 커밋 원칙

- 커밋은 학습 단위로 작게 만든다.
- 앱 코드만 바꾸고 문서를 빼먹지 않는다.
- 문서만 바꾼 경우에도 왜 바꿨는지 커밋 메시지에 드러낸다.

커밋 메시지 예시:

- `docs: 플러터 학습 계획 추가`
- `feat: 홈 피드 화면 뼈대 추가`
- `refactor: 게시글 카드 위젯 분리`
- `test: 게시글 좋아요 토글 테스트 추가`

## 다음 요청 처리 규칙

사용자가 아래처럼 말하면 이 문서를 기준으로 이어간다.

- "오늘 공부하자"
- "다음 작업 해줘"
- "어제 하던 거 이어서"
- "이번 단계 설명해줘"
- "커밋하자"

이때 에이전트는 먼저 진행 로그를 확인하고, 문서에 기록된 다음 작업에서 출발한다.
